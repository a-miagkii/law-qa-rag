# Веб-прототип: `law_qa_rag.web.app`

Веб-прототип — server-rendered интерфейс поверх текущего RAG pipeline. Он использует retrieval и LLM-настройки из `settings.yaml`, GigaChat provider из `src/law_qa_rag/llm` и сохраняет успешные ответы в PostgreSQL.

## Запуск

Сначала должен быть подготовлен корпус:

1. parsed JSON и chunks из `docs/1_parsing.md` и `docs/2_chunking.md`;
2. загруженные `acts` и `chunks` из `docs/3_load_corpus.md`;
3. рассчитанные embeddings из `docs/4_embed_chunks.md`, если retrieval использует `dense` или `weighted_hybrid`.

Локальные переменные лежат в `.env`:

```bash
cp .env.example .env
# Заполните POSTGRES_PASSWORD, SESSION_SECRET и GIGACHAT_CREDENTIALS.
```

Web-приложение автоматически читает `.env`. Если `DATABASE_URL` не задан, он собирается из `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_DB`, `POSTGRES_USER` и `POSTGRES_PASSWORD`.
При SSL timeout или self-signed certificate для локальной разработки можно поставить `GIGACHAT_VERIFY_SSL_CERTS=false` и увеличить `GIGACHAT_TIMEOUT`.

Для существующей БД, созданной до web-этапа, один раз примените миграции:

```bash
.venv/bin/python scripts/db/apply_migration.py sql/migrations/001_answers_web_fields.sql
.venv/bin/python scripts/db/apply_migration.py sql/migrations/002_auth_feedback.sql
```

Первая миграция добавляет к таблице `answers` web/retrieval-поля:

- `needs_clarification`;
- `retrieval_method`;
- `retrieved_chunk_ids`;
- `dropped_chunk_ids`.

Вторая миграция добавляет к `users` поля локальной авторизации и гарантирует наличие таблицы `feedback`.

Запуск:

```bash
.venv/bin/python -m uvicorn law_qa_rag.web.app:app \
  --host 127.0.0.1 \
  --port 8000 \
  --reload
```

Откройте `http://127.0.0.1:8000/`.

## Авторизация

В веб-интерфейсе используется минимальная локальная авторизация. В шапке неавторизованный пользователь видит одну кнопку «Войти». Она открывает модальное окно «Авторизация» с двумя вкладками:

- «Вход»;
- «Регистрация».

Регистрация создает пользователя в таблице `users` и сразу открывает сессию. Логин хранится в `users.external_uid`; это может быть email или простой username. Пароль хранится не в открытом виде, а как PBKDF2-SHA256 hash в `users.password_hash`.

Session хранится через `SessionMiddleware`. Секрет берется из `SESSION_SECRET`; если переменная не задана, используется dev-значение только для локального запуска.

Это не промышленная система управления доступом: нет ролей администратора, восстановления пароля, подтверждения почты, внешних OAuth-провайдеров и разграничения прав доступа к корпусу.

## Endpoint `/ask`

`POST /ask` принимает вопрос двумя способами, но требует авторизованного пользователя.

HTML form:

```bash
curl -i \
  -X POST \
  -F "question=Что такое водные объекты общего пользования?" \
  http://127.0.0.1:8000/ask
```

Если пользователь не вошел, HTML form получает `303 See Other` на главную страницу с открытым окном авторизации и сообщением: «Чтобы задать вопрос, войдите в систему или зарегистрируйтесь.»

Для авторизованного пользователя успешный ответ — `303 See Other` на `/answers/{answer_id}`.

JSON:

```bash
curl \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"question":"Что такое водные объекты общего пользования?"}' \
  http://127.0.0.1:8000/ask
```

Если пользователь не вошел, JSON-запрос получает `401`. Для авторизованного пользователя endpoint возвращает `answer_id`, `answer`, `used_chunk_ids`, `answer_citations`, retrieval metadata и `latency_ms`.

В обычном web-сценарии `queries.user_id` всегда равен id текущего авторизованного пользователя. Технический пользователь `external_uid=local-web` оставлен только для совместимости, отладки и старых данных.

## Страницы

- `/` — поле вопроса, кнопка отправки и примеры вопросов. Без входа кнопка предлагает авторизоваться.
- `/answers/{answer_id}` — вопрос, ответ, предупреждение при нехватке контекста, список цитат и форма оценки.
- `/sources/{act_id}?answer_id=...` — реквизиты акта и фрагменты этого акта, процитированные фрагменты подсвечены.

- `/profile` — профиль пользователя, кнопка выхода и история его вопросов.

Страница ответа доступна только владельцу ответа. Если пользователь не вошел, он перенаправляется на главную страницу с окном авторизации. Если вошел другой пользователь, сервер возвращает `403`.

Страница источника с параметром `answer_id` также проверяет, что ответ принадлежит текущему пользователю.

Профиль доступен только авторизованному пользователю. В истории профиля показываются только вопросы текущего пользователя: дата и время, текст вопроса, ссылка на сохраненный ответ, количество цитат и признак «требует уточнения».

На страницах выводится дисклеймер: ответ строится только по редакциям документов, загруженным в локальный корпус, и актуальность нужно проверять по официальному источнику.

## Оценка ответа

На странице ответа есть блок «Оценка ответа».

Авторизованный владелец ответа может:

- поставить оценку от 1 до 5;
- добавить комментарий;
- повторно отправить форму, чтобы обновить существующую запись.

Сохранение выполняется через `UPSERT` по паре `(answer_id, user_id)`, поэтому дубликаты feedback для одного ответа и пользователя не создаются. После сохранения пользователь возвращается на ту же страницу с сообщением «Спасибо, оценка сохранена».

`feedback` используется как источник пользовательской оценки качества ответа при опытной эксплуатации прототипа.

## Сохранение в БД

Успешный запуск сохраняет:

- `queries.question` и `queries.normalized_question`;
- `queries.user_id`: текущий авторизованный пользователь;
- `answers.answer_text`, `llm_model`, `prompt_version`, `latency_ms` и retrieval metadata;
- `answer_citations.chunk_id`, `rank`, `relevance_score`, `quote`;
- `feedback.rating` и `feedback.comment`, если пользователь оценил ответ.

Перед сохранением приложение не меняет схему БД. Для существующей БД schema update выполняется явно через SQL-миграции.

Ошибки GigaChat или БД показываются пользователю, но в v1 отдельно не логируются.
