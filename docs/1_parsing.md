# Парсинг: `scripts/parse_docs.py`

`parse_docs.py` преобразует `.doc`-файлы, экспортированные с pravo.gov.ru как HTML, в нормализованные JSON-файлы для последующего чанкинга.

Запуск:

```bash
.venv/bin/python scripts/parse_docs.py data/raw_codes data/parsed_json
```

Результат:

- отдельный `<source>.json` для каждого нормативного акта;
- `manifest.json` с метаданными акта, количеством nodes и предупреждениями парсера.

## Фильтрация входных файлов

Парсер обрабатывает только `.doc`-файлы. Временные файлы офисных редакторов, например `~$*.doc`, и скрытые файлы пропускаются до парсинга. Поэтому они больше не создают пустые акты с `unknown-date`.

Если валидных `.doc`-файлов нет, скрипт завершается с ошибкой.

## Формат parsed JSON

У каждого parsed JSON два верхнеуровневых поля:

```json
{
  "act": {},
  "nodes": []
}
```

`act` содержит стабильные метаданные документа:

- `canonical_key`;
- `act_kind`;
- `doc_type`;
- `title`;
- `doc_number`;
- `doc_date`;
- `official_text_kind`;
- `edition_as_of`;
- `edition_note`;
- `status`;
- `has_future_editions`;
- `source_file`;
- `source_system`.

`nodes` это упорядоченный поток структурных и текстовых узлов. Структурные узлы: `part`, `section`, `subsection`, `chapter`, `paragraph_group`, `article` и общий `heading`. Текстовые узлы: `paragraph` и `preamble`.

Каждый node хранит:

- `order`;
- `node_type`;
- `text`;
- `raw_text` для текстовых узлов, где важно сохранить исходную нумерацию;
- `source_anchor`;
- `article_no`;
- `clause_no`;
- `structure_ref`;
- `context`.

## Извлечение метаданных

Парсер больше не зависит только от фиксированных позиций строк в блоке `topText`. Он ищет строки по содержанию:

- заголовок документа определяется по виду `Кодекс Российской Федерации от DD.MM.YYYY № ...`;
- строка редакции определяется по словам `Редакция`, `актуаль` или формулировкам об утрате силы;
- тип официального текста определяется по слову `официальный`.

Это делает парсинг устойчивее, если HTML-экспорт добавит или уберет строку в верхнем блоке.

## Нормализация текста

Перед записью JSON парсер нормализует юридический текст:

- схлопывает повторяющиеся пробелы и неразрывные пробелы;
- восстанавливает номера статей, разбитые superscript-span, например `7 .1 -> 7.1`;
- удаляет редакционные span-блоки;
- сохраняет юридически значимые маркеры утраты силы коротким текстом, например `Пункт утратил силу.`;
- сохраняет нумерацию пунктов в `raw_text`, чтобы чанкинг не терял `1.`, `2)`, `12.1)` и похожие маркеры.

## Валидация

Для каждого распарсенного акта manifest содержит поле `warnings`. Сейчас фиксируются такие предупреждения:

- нет названия акта;
- нет даты или номера акта;
- не найдено ни одного юридического node.

Предупреждения не останавливают весь запуск. Они нужны, чтобы подсветить подозрительные исходные файлы.
